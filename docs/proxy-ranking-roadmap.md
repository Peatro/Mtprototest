# Proxy Ranking — Roadmap

Эволюция алгоритма выдачи прокси от статического ранжирования к адаптивному.

---

## Этап 1 — Per-session blacklist + segment overrides (текущий)

**Цель:** устранить thrash-паттерн и убрать заведомо нерабочие прокси из выдачи
конкретного сегмента без изменения схемы БД.

**Что делает:**
- Per-session blacklist: прокси, показанные в текущей сессии пользователя, исключаются
  из кандидатов при следующем запросе. Сессия = clientKey + 30-минутное окно.
- Segment overrides: задаются в конфигурации как пары `(proxy_id, country, os, multiplier)`.
  `multiplier = 0` → полное исключение. `multiplier < 1` → перемещение в конец очереди.
- Decision logging: каждое решение алгоритма пишется в структурированный лог.

**Инфраструктурные требования:**
- In-memory кеш (ConcurrentHashMap, уже доступен)
- Конфигурация в application.yml

**Метрика успеха:**
- Среднее число попыток на сессию снижается (PostHog: среднее `next_proxy_clicked` per session)
- Прокси 678, 776, 883, 1033 полностью исчезают из выдачи RU/Windows

**Оценка сложности:** S (1–2 дня)

---

## Этап 2 — Per-segment scoring

**Цель:** построить полную матрицу `proxy × segment` и автоматически учитывать её
при ранжировании без ручных overrides.

**Что делает:**
- Запускает фоновый агрегатор PostHog HogQL (или отдельный ETL), который считает
  `likely_worked` rate по каждой паре `(proxy_id, country, os)` за скользящее окно 7 дней.
- Результат записывается в таблицу `proxy_segment_stats (proxy_id, country, os, success_rate, sample_size, updated_at)`.
- При ранжировании `score` корректируется: `effective_score = score * segment_multiplier`,
  где `segment_multiplier` вычисляется из `success_rate` с учётом `sample_size`
  (Wilson score / Bayesian smoothing при малых выборках).

**Инфраструктурные требования:**
- Новая таблица `proxy_segment_stats` (Liquibase)
- Фоновый @Scheduled агрегатор или внешний ETL pipeline
- PostHog Export API или HogQL batch endpoint

**Метрика успеха:**
- `likely_worked` rate в топ-3 выдаче вырастает ≥10 п.п. по каждому крупному сегменту
- Overrides из Этапа 1 можно отключить (алгоритм сам вытесняет аутсайдеров)

**Оценка сложности:** M (1–2 недели)

---

## Этап 3 — Composite score из нескольких сигналов

**Цель:** учитывать все доступные поведенческие сигналы, а не только `likely_worked`.

**Сигналы для композитного скора:**

| Сигнал | Вес (прим.) | Интерпретация |
|--------|------------:|---------------|
| `likely_worked` | 0.45 | Пользователь кликнул и не вернулся |
| `worked_clicked` | 0.20 | Явный фидбек «сработало» |
| `time_to_return` | 0.20 | Чем быстрее вернулся — тем хуже прокси |
| `next_proxy_clicked` | 0.15 | Явный отказ от прокси |

**Что делает:**
- Хранит per-session временны́е метки событий в `proxy_session_signals`.
- После завершения сессии (timeout или явный сигнал) агрегирует в `proxy_segment_stats`.
- Composite score пересчитывается раз в час по скользящему 7-дневному окну.

**Инфраструктурные требования:**
- Сигнальная таблица (Liquibase)
- Агрегатор (Scheduled или stream-processing)
- Kafka (опционально, для надёжной доставки сигналов)

**Метрика успеха:**
- A/B тест: composite score vs likely_worked-only → composite даёт ≥5% лучший first-attempt success rate

**Оценка сложности:** L (2–4 недели)

## Этап 4 — Multi-armed bandit (Thompson sampling)

**Цель:** автоматически балансировать exploration/exploitation без ручного тюнинга весов.

**Что делает:**
- Каждый прокси в каждом сегменте моделируется Beta-распределением Beta(α, β),
  где α = successful_impressions + 1, β = failed_impressions + 1.
- При выборе прокси из кандидатов делается Thompson sample: для каждого прокси
  сэмплируется θ из его Beta, выбирается прокси с максимальным θ.
- α и β обновляются по `likely_worked` сигналу в реальном времени или по батчу.
- Exploration встроена: прокси с малой выборкой имеют широкое распределение и
  получают шансы на показ автоматически.

**Инфраструктурные требования:**
- Хранение (α, β) per (proxy, segment) — Redis или PostgreSQL
- Сэмплирование в realtime (< 1 мс) → желательно Redis + in-process cache
- Сигнальный pipeline (Этап 3 prerequisite)

**Метрика успеха:**
- Regret (упущенные конверсии vs оракул) снижается по сравнению с Этапом 3
- Новые прокси автоматически получают exploration без ручного вмешательства

**Оценка сложности:** XL (4–8 недель, включая инфраструктуру)

## Этап 5 — Targeted route для «невосстанавливаемых»

**Цель:** не мучить пользователей, для которых ни один из прокси не работает, —
вместо этого собрать диагностическую информацию и дать альтернативный путь.

**Что делает:**
- После N неудачных попыток в сессии (next_proxy_clicked ≥ N или явный failed_clicked ≥ N)
  UI переключается в режим «диагностика»:
  - Показывается форма: «Какой у тебя провайдер / страна?»
  - Опционально: предлагается установить VPN вместо прокси
  - Альтернативные каналы: ссылка на Telegram-канал с актуальными прокси
- Диагностические данные (провайдер, ISP, регион) попадают в отдельный PostHog event
  `unrecoverable_session` и помогают выявить блокировки на уровне ISP.

**Инфраструктурные требования:**
- Frontend: новое UI-состояние (minimal JS)
- Backend: endpoint для приёма диагностики (опционально)
- Пороговые настройки в конфигурации

**Метрика успеха:**
- Снижение доли сессий с ≥ 5 `next_proxy_clicked` без `open_telegram_link`
- Рост числа диагностических отчётов → данные для улучшения Этапа 2

**Оценка сложности:** M (1–2 недели)
